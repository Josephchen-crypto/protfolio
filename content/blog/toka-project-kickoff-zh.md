# Toka项目启动记

这个项目正式启动是在2023年8月份。我是7月份入职，用一个月熟悉公司的规章制度和流程。期间也参与了一些其他项目组的小功能开发，同时了解各部门的协作方式——从功能调研、开发、测试、bug修复到正式上线，每个环节都有对应的负责人。

我其实是个半路出家的程序员，以往的工作很少能接触到这么正规的研发流程。大部分公司由于资金和资源限制，只能快速迭代快速试错，没法像大公司这样在每个环节都保证质量。

8月份，我被安排进了Toka这个新项目。项目启动前有两周左右的动员会议，了解项目背景和甲方情况。这些会议资料，据我估计甲方早在一个月前就开始准备了——应该在墨西哥和甲方进行了充分沟通，了解企业背景、具体业务，然后才有了PRD和相关启动会议。

动员会上介绍了甲方信息、里程碑划分、每个阶段的目标，以及各部门的负责人。我被安排主导Android端开发，iOS由公司一位三年的老员工负责，他同时也带我熟悉初期的项目开发流程。

## 开发过程中的规章制度

出乎意料的是，公司对开发的规定比我之前待过的公司要规范得多，当然流程也繁杂得多。我从中意识到这些规范对项目研发的作用——虽然最初学起来相当笨拙，但熟悉起来只是时间问题。

就拿git仓库管理来说，就让我头疼了一阵。项目初期一切都不是我熟悉的那套简单流程。首先是权限申请，公司有专门的git管理平台，操作仓库需要申请和审批——也就是说仓库虽然归我主导，但代码不是随便能动的。

更让我头疼的是，项目启动不久我就遇到了一次误操作需要回退代码，这同样需要管理员授权，而且还要先通报项目经理向上反馈。任何试图绕过这套机制的操作不仅无效，反而会把问题搞大。所以无形之中，每次提交和合并分支都得非常仔细地检查，压力不小。

## 项目仓库操作——分支命名规范

公司在这方面积累很深，分支命名有严格格式：

```
(feature/bugfix)-(version<month(a/b)>)-(feature_name)-(task_id)-(dev_name)
```

例如：`feature-5a-aml_pin-123423-joseph`

含义如下：

- **feature / bugfix**：一眼区分是功能开发还是bug修复。
- **版本周期**：每月分前半月a和后半月b，每个季度的大功能被拆成小任务分配给开发人员。新功能起新分支，需要备注当月版本——5月份上半月的任务就是5a。
- **功能名称**：公司那套OA系统和git仓库打通了。在任务单上点击创建分支，系统会自动根据任务信息命名。不过还是要手动补上版本信息。
- **任务单号**：系统自动补充，方便完成后快速找到对应单据，改为待测试状态，之后由CI/CD自动打包验证。
- **开发人员**：分支必须带上名字缩写，多人协作时用于区分。

## 加班与学习

虽然这套流程耗时，和快节奏的开发相违背，但误操作带来的损失更大。这也间接导致了一个问题：项目只有我一个人的时候，既要学习公司的开发规范，又要赶被压缩到极短的开发周期，我被逼无奈只能每晚疯狂加班。哪怕下班回到家，也必须坐在电脑前疯狂敲键盘完成指定的功能。
